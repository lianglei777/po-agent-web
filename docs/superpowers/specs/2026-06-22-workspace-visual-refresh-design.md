# Po Agent Web 工作台视觉升级设计

日期：2026-06-22
状态：已确认，等待实现计划

## 1. 目标

在不改变现有三栏工作流、业务逻辑和操作路径的前提下，提高 Po Agent Web 的设计完成度、长时间使用舒适度与信息扫描效率。

新版应融合：

- Linear 的秩序与精密感；
- Raycast 的命令中心效率；
- Cursor 的项目上下文工作台属性。

最终产品仍然是安静、精准、开发者可控的工具，而不是营销式 AI 聊天界面。

## 2. 已确认的视觉方向

采用一套结构一致、独立调色的双主题系统：

- Light：以 `Adaptive Workbench` 为基础，使用冷灰中性色与低饱和松绿色焦点色。
- Dark：以 `Deep Focus` 为基础，使用炭黑中性色与明亮薄荷青焦点色。

两套主题不是简单反色。每套主题分别校准背景层级、文字对比、边框强度和交互状态，但共享相同的语义 token、组件结构、尺寸、圆角与动效规则。

## 3. 设计原则

### 3.1 信息密度优先

保留会话、文件、聊天和文件预览组成的三栏工作台。提升层级与可扫描性，不通过减少关键内容制造“简洁”。

### 3.2 安静表面，明确状态

默认界面保持低对比、低色度。强调色仅用于：

- 当前选择；
- 键盘焦点；
- 主要操作；
- Agent 运行状态；
- 上下文用量等需要关注的状态。

强调色不用于装饰、大面积背景或普通图标。

### 3.3 结构先于装饰

高级感主要来自稳定比例、精确对齐、文字层级、边界关系和完整状态，而不是渐变、玻璃模糊、大阴影或夸张圆角。

### 3.4 长时间工作舒适度

Light 主题避免大面积纯白与暖米色；Dark 主题避免纯黑、霓虹和过强边框。正文、占位文字与次级信息均需满足可读性要求。

### 4.3 语义状态色

错误、警告、成功和信息状态继续使用独立语义色。它们必须同时通过文字、图标、边框或形状表达，不能仅依赖颜色。

## 5. 字体与文字层级

- UI、导航、按钮、表单和聊天内容使用统一的无衬线字体栈。
- 路径、模型参数、上下文数据和代码使用现有等宽字体栈。
- 衬线展示字体只保留在欢迎标题等极少数空状态场景；如果实际浏览器效果与整体工具气质割裂，实现时应改为统一无衬线标题。
- 主界面不使用流式超大字号。欢迎标题建议为 `28–32px`，其余层级使用紧凑固定尺度。
- 中文与英文必须分别检查字重、行高、截断和密度。

## 6. 布局与组件设计

### 6.1 工作台框架

保留当前可折叠、可调整尺寸的三栏结构：

- 左侧：项目选择、会话与文件浏览；
- 中间：聊天、Agent 执行过程与 Composer；
- 右侧：文件内容，后续可扩展为上下文检查器。

本次视觉升级不引入双层侧边导航，不把会话和文件改造成互斥 Tab，也不改变现有面板操作方式。

### 6.2 顶部工具栏

- 保持紧凑高度，统一图标按钮尺寸与边界规则。
- 左侧保留侧栏、主题和语言操作。
- 右侧保留模型、技能和文件面板操作。
- 当前项目、Agent 状态或上下文用量只在数据真实可靠且不会造成拥挤时显示。
- 使用分组、间距和 tooltip 建立层级，不使用装饰性 pill 堆叠。

### 6.3 左侧栏

- 项目选择器作为侧栏顶部的稳定锚点。
- 会话与文件浏览继续上下分区，并保留可拖动分隔。
- 会话选中状态使用背景、文字权重和轻量强调色共同表达。
- 文件树保持高密度；增加清晰的 hover、focus-visible 与当前文件状态。
- 分区标题降低视觉噪声，不使用过宽字距和过强大写标签。

### 6.4 欢迎空状态

空状态从单一标题升级为简洁的任务入口：

- 产品或工作区标识；
- 结果导向的主标题；
- 一句说明当前工作区能力的辅助文字；
- 四个常用起点。

常用起点应使用轻量边界行或紧凑按钮，不使用带大阴影的营销卡片。点击后仅填充 Composer，不自动发送。

### 6.5 Composer

Composer 是主要操作中心，但仍属于工作台结构，不做悬浮玻璃卡片。

- 使用完整的 `border-strong` 边界和明确的 focus-within 状态。
- 保持现有文本输入、模型选择、附件、思考、工具、压缩、发送和运行中操作。
- 主操作使用主题 accent；禁用态保持低权重且可辨认。
- 模型与高频操作位于第一层；思考、工具和压缩保持第二层。
- 运行、重试、错误、停止、排队和引导状态需要形成一致状态带。

### 6.6 右侧面板

- 继续默认收起；只有用户主动打开文件或面板时占用空间。
- 无文件时保持简洁空状态，不强行填充虚构的上下文数据。
- 文件标题、关闭操作和正文区域使用与主工作台一致的层级。
- 将右侧面板扩展为 Context Inspector 属于后续功能提案，不包含在本次视觉升级中。

## 7. 形状、边界与阴影

- 小图标控件使用 `4px` 圆角。
- 按钮、输入框和 Composer 使用 `6px` 圆角。
- 浮层和对话框使用 `8px` 圆角。
- 普通面板、列表和 Composer 不使用宽软阴影。
- 下拉、Tooltip、Popover 和 Dialog 可使用低强度 `0 2px 8px` 阴影。
- 面板结构主要依靠背景层次与边界，不依靠卡片化。

## 8. 动效与反馈

- 状态变化使用 `150–220ms`。
- hover、focus、selected、panel 展开和尺寸变化只动画颜色、透明度、短距离位移或必要尺寸。
- Agent 运行状态可使用轻微脉冲；在 reduced motion 下改为静态状态点。
- 不增加页面入场编排、背景动画、弹跳或装饰性转场。

## 9. 响应式行为

- 桌面：保留三栏与可调宽度。
- 中等宽度：右侧面板按需覆盖或隐藏，中央工作区优先。
- 移动端：左侧栏使用现有抽屉模式；文件面板占据主内容区域。
- Composer 在窄屏将低频设置收进菜单，保留附件、模型和发送等主要操作。
- 中英文以及 Light/Dark 均需验证桌面、中等和移动宽度。

## 10. 无障碍要求

- 正文和占位文字目标对比度不低于 `4.5:1`。
- 大文字与非文本控件边界目标不低于 `3:1`。
- 所有交互控件保留可见的 `focus-visible` 状态。
- 选中、错误、警告和运行状态不得仅依赖颜色。
- 保留语义 HTML、标签、键盘操作、Tooltip 与 reduced motion 支持。

## 11. 实现范围

本轮包含：

- 全局 Light/Dark 语义 token；
- 基础组件状态；
- 工作台背景与面板层级；
- 顶部工具栏；
- 左侧会话与文件区；
- 欢迎空状态；
- Composer；
- 文件面板空状态与标题区；
- 中英文与响应式视觉验证。

本轮不包含：

- 新增后端能力；
- 修改 API 契约；
- 新增任务系统；
- Command Palette；
- 右侧 Context Inspector 的真实数据能力；
- Diff 编辑器；
- 会话与文件的信息架构重构；
- 引入新的 UI 或动效依赖。

## 12. 验收标准

- Light 主题呈现 B 方案的冷静、耐看与清晰层级。
- Dark 主题呈现 C 方案的沉浸感，但不出现纯黑、霓虹或游戏化观感。
- 两套主题在结构、组件和品牌气质上明显属于同一产品。
- 用户能快速识别当前项目、当前会话、主输入入口、文件区和面板操作。
- Accent 覆盖面积克制，静止界面仍以中性色为主。
- Composer 是明确的主操作入口，但不脱离工作台成为装饰性浮层。
- 所有现有核心操作路径和业务行为保持不变。
- Light/Dark、中文/英文、桌面/移动均无明显截断、对比度或焦点问题。
- `npm run check` 通过。
- 涉及 Next.js 生产样式与渲染，`npm run build` 通过。
