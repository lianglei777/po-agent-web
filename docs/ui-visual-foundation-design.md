# Po Agent Web 视觉基础重构设计规格

日期：2026-06-18

状态：待用户最终审阅

实施状态：未授权，不得修改产品代码

## 1. 背景

Po Agent Web 当前已经具备稳定的三栏工作区、会话管理、聊天、文件预览、模型配置和技能配置能力。本轮工作不重构功能逻辑，也不重新设计布局或操作路径，而是为现有界面建立更明确、更一致的视觉语言。

视觉方向参考 `docs/design-prompt.md` 中的三部分：

1. 柔化黑白色彩；
2. 衬线展示字体；
3. 线性秩序。

原提示词包含大量面向品牌页或营销页的规则，例如巨型标题、全直角、纹理背景、粗重分区线、Pricing、Testimonials、Hero 和 CTA。这些规则不适合高密度开发者工具，因此不直接应用。

本设计采用 **Token-first** 路径：先建立设计 Token，再统一基础组件，最后处理功能界面的局部差异。

## 2. 已确认的设计目标

- 保持现有功能、数据、接口与业务行为不变。
- 保持现有布局、信息架构和主要操作路径不变。
- 保持工作区的信息密度和开发者工具属性。
- 让浅色与深色主题拥有一致但独立调校的视觉层级。
- 使用柔化黑白、受控的衬线标题和分层边界建立辨识度。
- 统一基础组件的默认、悬停、焦点、选中、禁用、加载和错误状态。
- 保持中文和英文界面的排版稳定。
- 保持键盘操作、焦点可见性、响应式和 reduced-motion 支持。
- 为重要删除操作和 Dialog 关闭行为建立一致的安全规范。

## 3. 非目标

本轮设计与首轮实现不包含：

- 重做工作区布局；
- 改变侧栏、聊天区或文件面板的位置；
- 将弹窗改造成抽屉或新页面；
- 改变功能入口、操作流程或数据结构；
- 新增业务功能；
- 引入装饰性背景、纹理、渐变或玻璃效果；
- 引入 ReactBits 或其他新增动效依赖；
- 使用大面积动画、粒子、磁吸、光标尾迹或动态背景；
- 将产品改造成编辑杂志、奢侈品网站或营销落地页。

## 4. 设计原则

### 4.1 工具优先

设计服务于任务，不与任务竞争。清晰、稳定、快速扫描优先于视觉戏剧性。

### 4.2 黑白不是绝对黑白

大部分表面使用近黑、冷白和中性灰。纯黑与柔白只用于关键操作、强边界和高优先级状态。

### 4.3 衬线字体是有限的强调

衬线字体只承担少量展示角色。导航、按钮、列表、表单、正文和数据仍使用高可读的 sans 或 mono。

### 4.4 线条表达结构

结构通过边界、分隔线和留白表达，而不是为每个元素增加卡片、阴影或完整外框。

### 4.5 一致性高于局部创意

共享组件和 Token 是视觉规则的唯一入口。功能组件不得随意添加新的颜色、圆角、阴影或过渡时长。

### 4.6 状态必须可辨识

视觉克制不能牺牲错误、成功、警告、运行、焦点和禁用状态的辨识度。

## 5. 设计系统架构

视觉系统分为四层：

1. **基础值层**：色值、字体、尺寸、圆角、阴影和时长。
2. **语义 Token 层**：canvas、panel、text、border、selected、destructive 等。
3. **基础组件层**：Button、Input、Dialog、Select、Card 等。
4. **功能界面层**：聊天、会话、文件、模型和技能界面。

依赖方向为：

`基础值 → 语义 Token → UI 基础组件 → 功能界面`

功能界面可以组合基础组件，但不应重新定义视觉系统。

## 6. 色彩系统

### 6.1 色彩策略

采用柔化黑白和中性灰阶：

- 浅色主题以冷白画布、纯白内容面和近黑文字为主；
- 深色主题以近黑画布、炭灰面板和柔白文字为主；
- 两套主题独立调校，不采用简单反相；
- 不设置装饰性品牌强调色；
- 低饱和语义色仅用于状态反馈。

### 6.4 语义色

保留低饱和语义色：

| 状态 | 浅色主题 | 深色主题 | 使用范围 |
| --- | --- | --- | --- |
| 错误/破坏性 | `#A63F38` | `#E07A73` | 删除、错误、停止 |
| 成功 | `#346B4D` | `#78B28F` | 保存成功、连接成功 |
| 警告/运行 | `#815B18` | `#D2A65A` | 警告、重试、运行状态 |

语义色规则：

- 只用于状态，不用于装饰或品牌表达；
- 大面积背景使用低透明度色调，文字和图标使用实色；
- 不允许只用颜色传递状态，必须配合文本、图标、边框或形状；
- 最终色值需分别在浅色和深色背景上验证 WCAG AA；
- 不同状态不得共用同一颜色。

### 6.5 对比度要求

- 正文和占位符对背景至少达到 4.5:1；
- 大文字至少达到 3:1；
- 焦点指示器和关键控件边界至少达到 3:1；
- `text-dim` 不得成为不可读的装饰灰；
- 禁用状态可以降低强调，但仍应保持标签可识别。

## 7. 字体系统

### 7.1 字体角色

| 角色 | 字体 |
| --- | --- |
| 拉丁展示字体 | Playfair Display |
| 中文展示字体 | Noto Serif SC |
| UI 正文与控件 | 现有 sans 字体栈 |
| 代码与元数据 | 现有 mono 字体栈 |

### 7.2 展示字体使用范围

衬线展示字体只用于：

- 欢迎页主标题；
- 当前会话主标题；
- Dialog 的大标题。

衬线字体不得用于：

- 按钮；
- 导航；
- 会话列表项；
- 文件树；
- 表单标签和输入内容；
- 状态提示；
- 模型和技能配置字段；
- 工具调用和日志；
- 长篇聊天正文。

### 7.3 字号与字重

- 欢迎页主标题：桌面端约 30–36px，移动端约 26–30px；
- 会话主标题：约 20–24px；
- Dialog 大标题：约 20–24px；
- 不使用 8xl、9xl 或营销式巨型字号；
- 展示字体优先使用 600 字重；
- 不使用过紧字距，最小不低于 `-0.03em`；
- 中文展示标题采用正常或轻微收紧字距，不进行拉丁式极端紧排。

### 7.4 字体加载

- 仅加载实际需要的字重；
- 优先通过 Next.js 字体能力加载；
- 避免将大范围中文字体字形无条件加入首屏；
- 必须提供稳定的 serif 回退栈；
- 字体加载前后不得造成明显布局跳动；
- 中英文混排必须保持基线、行高和视觉重量协调。

## 8. 线性秩序

### 8.1 三级边界

#### 结构深线

用于：

- 主面板之间的边界；
- 侧栏与内容区边界；
- 顶部栏与主体边界；
- 聊天输入区与消息区边界；
- 关键选中状态。

结构深线使用 `border-strong`，少数最关键位置可使用 `border-emphasis`。

#### 内容浅线

用于：

- 列表项分隔；
- 字段组分隔；
- 消息中的工具调用区；
- 面板内部标题区；
- 表单分组和元数据区域。

内容浅线使用 `border-subtle`。

#### 浮层完整框

用于：

- Dialog；
- Dropdown；
- Select 内容层；
- Tooltip；
- Popover；
- 其他脱离文档流的浮层。

浮层使用完整外框，并可配合受控阴影。

### 8.2 禁止事项

- 不给每个列表项、按钮和信息块都增加完整边框；
- 不使用粗侧边彩条作为状态装饰；
- 不使用大面积 4px 或 8px 黑线切割所有内容；
- 不使用重复线条、网格、噪点或纸张纹理背景；
- 不使用边框加宽模糊阴影的“幽灵卡片”组合。

## 9. 圆角与阴影

### 9.1 圆角

| 角色 | 建议值 |
| --- | --- |
| 小型控件 | 4px |
| 输入框、按钮、列表内控件 | 6px |
| Dialog、Popover、Dropdown | 8px |
| Badge、状态点、开关轨道 | pill 或 circle |

规则：

- 普通容器不超过 8px；
- 聊天输入区不再使用夸张的 20px 圆角；
- 不使用 12–32px 的大面积圆角卡片；
- 圆角必须由 Token 提供，禁止功能组件随意填写数值。

### 9.2 阴影

阴影只用于真正浮动的元素：

- Dialog；
- Dropdown；
- Select；
- Tooltip；
- Popover。

建议阴影特征：

- 短距离；
- 低模糊；
- 低透明度；
- 不带彩色辉光；
- 深色主题中主要依赖边框和表面层级，阴影更弱。

页面、面板、列表和普通内容容器默认无阴影。

## 10. 状态规范

每个交互组件必须覆盖以下状态：

1. Default
2. Hover
3. Focus-visible
4. Active/Pressed
5. Selected
6. Disabled
7. Loading
8. Error

### 10.1 Hover

- 仅做轻微明度变化；
- 不改变尺寸、边框宽度或布局；
- 不使用位移、放大或强阴影。

### 10.2 Focus-visible

- 使用清晰轮廓或焦点环；
- 浅色主题使用近黑，深色主题使用柔白；
- 不得只依赖颜色变浅；
- 焦点不得被 `overflow: hidden` 裁切。

### 10.3 Active 和 Selected

- Active 比 Hover 更明确；
- Selected 通过背景、文字权重和边界共同表达；
- 不能只依赖彩色指示条；
- 不造成布局位移。

### 10.4 Disabled

- 禁止交互；
- 光标和 ARIA 状态正确；
- 文本仍可识别；
- 不与 Loading 混淆。

### 10.5 Loading

- 阻止重复提交；
- 保持控件尺寸稳定；
- 使用明确的进行中文案或克制进度图标；
- 长内容加载优先使用 Skeleton。

### 10.6 Error

- 错误出现在相关操作附近；
- 使用低饱和红、文字和必要图标；
- 不清除用户输入；
- 提供恢复或重试路径。

## 11. 基础组件规范

### 11.1 Button

- Primary：近黑实底、柔白文字；
- Secondary：中性表面和清晰边界；
- Outline：透明或面板背景、结构边界；
- Ghost：无默认边框，Hover 使用轻微表面色；
- Destructive：仅危险操作使用语义红；
- 图标按钮必须有可访问名称；
- 禁止通过阴影让所有按钮显得悬浮。

### 11.2 Input 和 Textarea

- 使用 6px 圆角；
- 默认边界清晰但不抢眼；
- Focus 使用焦点环或边界强化；
- Error 同时改变边界并显示错误文本；
- Placeholder 必须满足可读性要求；
- 聊天输入容器可以是组合控件，但内部层级仍使用共享 Token。

### 11.3 Select、Dropdown 和 Popover

- 使用完整外框；
- 使用 8px 以内圆角；
- 仅浮层使用受控阴影；
- 保持现有 Radix 键盘行为；
- 选中项、悬停项和禁用项必须有不同状态。

### 11.4 Card 和内容容器

- 不把所有内容都转化为 Card；
- 只有语义独立或需要从背景中分离的内容使用 Card；
- 普通分组优先使用留白和分隔线；
- Card 默认不使用装饰性阴影。

### 11.5 Badge 和状态标记

- 可使用 pill；
- 状态 Badge 使用语义色；
- 中性 Badge 使用灰阶；
- 文案必须明确，不能只显示颜色点。

### 11.6 Dialog

- 使用完整外框、8px 以内圆角和受控阴影；
- 大标题使用展示衬线字体；
- 正文、字段、按钮继续使用 sans；
- 必须有可见的明确退出动作；
- 遵循第 13 节的安全交互规范。

## 12. 功能界面应用

### 12.1 聊天页

聊天页是首个视觉验收样板。

验证范围：

- 欢迎页标题；
- Starter prompts；
- 用户消息；
- 助手消息；
- 工具调用；
- 错误和重试状态；
- 流式生成状态；
- 聊天输入区；
- 模型、Thinking 和 Tools 控件；
- 文件附件；
- Minimap；
- 浅色和深色主题；
- 桌面和移动端。

不改变：

- 消息结构；
- 发送、停止、排队和 steer 行为；
- SSE 生命周期；
- 输入快捷键；
- 文件拖放；
- Minimap 逻辑。

### 12.2 会话侧栏

- 保持现有层级和宽度逻辑；
- 使用结构深线分隔侧栏；
- 使用内容浅线组织会话和文件树；
- 选中态采用背景、文字和边界组合；
- 删除会话继续使用现有行内二次确认。

### 12.3 文件面板

- 保持现有打开、关闭和调整宽度逻辑；
- 文件路径和代码继续使用 mono；
- 空状态标题可使用 sans，不强制使用展示衬线；
- 预览容器优先使用边界和留白，不增加卡片阴影。

### 12.4 模型配置

- 保持现有功能和表单结构；
- Dialog 大标题使用展示衬线；
- Provider、Model、OAuth 和 API Key 表单继续使用 sans/mono；
- 删除 Provider、移除 Model、移除 API Key 等重要操作使用确认 Dialog；
- 未保存内容的退出遵循安全关闭规范。

### 12.5 技能配置

- 保持现有列表、详情、搜索和安装流程；
- Dialog 大标题使用展示衬线；
- 安装和切换状态使用语义色与明确文案；
- 不增加装饰性动效。

## 13. 安全交互规范

### 13.1 重要删除和不可逆操作

以下操作必须二次确认：

- 删除 Provider；
- 移除 Model；
- 移除 API Key 或断开认证；
- 其他会造成持久数据丢失、凭据移除或不可逆变化的操作。

例外：

- 删除会话保留现有行内二次确认，不改为 Dialog。
- 移除未提交的临时附件不视为重要删除。
- 清除纯本地、可即时恢复的临时 UI 状态不要求确认。

### 13.2 确认内容

确认界面必须说明：

- 将操作的对象名称；
- 操作造成的影响；
- 何时生效；
- 是否可以恢复；
- 如果尚未保存，需明确说明保存后才会生效。

危险按钮使用具体动词，例如：

- 删除提供商；
- 移除模型；
- 移除 API Key；
- 断开认证。

禁止仅使用含义模糊的“确定”作为危险按钮文案。

### 13.3 危险操作焦点与执行状态

- 默认焦点放在取消按钮或安全动作上；
- 不默认聚焦危险按钮；
- 执行期间禁用重复提交；
- 显示进行中状态；
- 失败时保持确认界面打开；
- 保留对象信息和错误上下文；
- 提供重试或取消路径。

### 13.4 Dialog 关闭规则

所有 Dialog：

- 禁止点击背景自动关闭；
- 禁止按 `Escape` 自动关闭；
- 只能通过可见的关闭、取消、保存、确认等动作退出；
- 必须至少提供一个明确的安全退出动作；
- 关闭按钮必须有可访问名称；
- 禁止把背景点击或键盘 Escape 作为唯一退出方式。

### 13.5 未保存内容

当 Dialog 存在未保存修改时：

- 点击关闭或取消不能直接丢弃内容；
- 必须提示是否放弃更改；
- 提示中说明哪些修改将丢失；
- 默认焦点放在“继续编辑”或等价安全动作；
- “放弃更改”使用明确文案；
- 保存成功后才允许按正常路径退出。

## 14. 动效规范

### 14.1 第一阶段

- 不安装 ReactBits；
- 不新增动效依赖；
- 保留必要的现有状态动效；
- 将常规过渡统一到 150–220ms；
- 优先动画 opacity、transform、颜色和短阴影；
- 不动画宽高等布局属性，面板调整宽度等既有交互除外；
- 拖拽、输入和连续滚动不得增加延迟。

### 14.2 Reduced Motion

- 所有非必要动效响应 `prefers-reduced-motion: reduce`；
- 降级为瞬时切换或非常短的淡入；
- Skeleton 和 Loading 状态不得造成明显闪烁；
- 内容默认必须可见，不能依赖动画完成后才显示。

### 14.3 ReactBits 后续评估

ReactBits 不属于本轮首轮实现。

视觉基础稳定后，若仍需要局部动效，应另行评估：

- 是否比 CSS 或现有 shadcn/Radix 动效更有价值；
- 需要哪些 peer dependencies；
- 客户端包体影响；
- Next.js SSR 和 hydration 行为；
- 组件卸载后的清理；
- reduced-motion 支持；
- 键盘和屏幕阅读器行为；
- 是否引入与产品气质冲突的装饰性表现。

只允许考虑状态反馈型动效，不允许动态背景、粒子、3D、磁吸、光标轨迹或营销式文字动画。

## 19. 验收标准

设计和实现完成需要同时满足：

- 功能逻辑、公共接口和数据结构不变；
- 布局、信息架构和主要操作路径不变；
- 视觉规则主要由 Token 和共享组件提供；
- 不出现未登记的颜色、字体、圆角、阴影和时长；
- 浅色和深色主题均层级清晰；
- 正文、占位符、焦点和状态满足 WCAG AA；
- 衬线字体只出现在已批准的展示位置；
- 中文和英文标题稳定，无明显布局跳动；
- 结构深线、内容浅线和浮层完整框使用一致；
- 普通内容不被过度卡片化；
- Dialog 不能通过背景点击或 Escape 关闭；
- 重要删除操作具有二次确认；
- 删除会话继续使用行内确认；
- 未保存内容不能被静默丢弃；
- Reduced Motion 生效；
- `npm run check` 通过；
- `npm run build` 通过；
- 浏览器关键流程验证通过；
- ReactBits 未被加入首轮实现。
